# RISC-V RV32I/RV64I面向Zynq-7020的高性能CPU架构设计方案

## 文档基础信息

- 设计语言：Verilog HDL（纯可综合语法，无冗余逻辑）

- 目标主频：100MHz保底，125MHz优化目标（Zynq-7020上板验证）

- 核心性能目标：100MHz下CoreMark≥**230Iter/s**，125MHz目标CoreMark≥**285Iter/s**

- 等效效能：≥2.3CoreMark/MHz

- 指令集：支持RV32I/RV64I参数化切换，**可选M扩展（RV32IM/RV64IM，硬件乘法器+除法器）**

- 核心特性：全参数化设计、标准AXI4-Lite总线、性能优先配置、局部历史分支预测、增强指令预取、哈佛BRAM存储结构

- 设计原则：面向Zynq-7020优先提升IPC与主频，允许使用更多LUT/FF/BRAM/DSP换取性能，只控制在器件合理预算内

- 对比基线：PicoRV32（~0.5 CoreMark/MHz）、ibex（~2.5 CoreMark/MHz），本设计目标是在Zynq-7020资源预算内接近ibex的单位频率性能，并显著提升相对PicoRV32的执行效率

## 一、设计目标与约束条件

### 1.1 核心设计目标

1. **性能目标**：100MHz主频下CoreMark跑分≥230Iter/s，125MHz优化目标≥285Iter/s，平均IPC≥0.85

2. **资源目标**：资源消耗服从性能目标，允许使用Zynq-7020上更多DSP、BRAM和LUT；资源验收以不超过器件合理预算为准

3. **指令集**：RV32I/RV64I位宽顶层参数一键切换，**M扩展可选实现（RV32M全8条指令；RV64M额外支持MULW/DIVW/DIVUW/REMW/REMUW）**

4. **架构**：**指令预取技术**、**局部历史分支预测器**、**最小CSR子集**、**基础异常处理**

5. **总线兼容**：标准AXI4-Lite从接口，适配SoC集成，支持上位机加载测试程序

6. **验证要求**：通过riscv-tests官方指令集测试、CoreMark性能基准测试

### 1.2 硬性约束条件

1. 资源占用：不再追求极限精简，优先保证CoreMark、IPC和时序目标；资源使用控制在Zynq-7020可部署范围内

2. 运算单元：启用M扩展时必须集成乘法器+除法器，RV32模式覆盖RV32M，RV64模式覆盖RV64M所需W后缀乘除法

3. 分支预测：采用**局部历史分支预测器（Local History Branch Predictor）**，分支在ID阶段提前解析，误预测惩罚为1周期

4. 取指优化：实现**指令预取缓存技术**，提升取指效率，减少流水线空等

5. 设计复用：全参数化逻辑复用，RV32I/RV64I切换保持同一套架构；允许不同位宽使用不同性能优化参数

6. 存储架构：双端口BRAM哈佛结构，CPU直连+AXI4-Lite调试隔离

7. 最小系统支持：Machine模式CSR子集 + 基础异常处理，满足CoreMark与riscv-tests需求

### 1.3 性能核算依据

```
目标 CoreMark/MHz = 230 / 100 = 2.3
目标平均 IPC ≥ 0.85

IPC损耗分解估算（基于CoreMark典型指令分布）：
  - 分支指令占比 ~20%，预测准确率93%，误预测惩罚1周期：0.20 × 0.07 × 1 = 0.014
  - Load-Use数据依赖占比 ~10%，每次插入1周期气泡：0.10 × 1 = 0.100
  - 串行除法指令占比 ~2%，平均额外开销8周期：0.02 × 8 = 0.160
  - 其余指令单周期完成，无额外开销
  理论IPC ≈ 1 / (1 + 0.014 + 0.100 + 0.160) ≈ 0.78（保守估算）
  实际优化后（更深预测表+增强预取+除法器提前终止+旁路覆盖）预期IPC ≈ 0.85~0.92
```

### 2.6 RV64I扩展参数说明

当`XLEN=64`时，自动生效以下变化：

- ALU运算通路扩展至64位，新增ADDW/SUBW/SLLW/SRLW/SRAW等W后缀指令译码
- 寄存器堆宽度扩展至64位，寄存器数量不变（x0~x31）
- 乘法器扩展至64×64位，额外消耗约2倍LUT或DSP资源
- Load/Store支持LW/LD/LWU等64位访存指令
- 立即数符号扩展至64位
- CSR寄存器宽度扩展至64位

## 三、总体架构

### 3.1 顶层模块化架构

```

│                                                         │

```

1. **CPU_TOP顶层模块**：全局参数、时钟复位、模块互联、AXI4-Lite接口封装

2. **5级流水线核心**：IF取指（含指令预取）、ID译码（含分支预测与提前解析）、EX执行、MEM访存、WB写回

3. **指令预取单元**：4级预取缓存，提前取指并覆盖顺序流与预测跳转目标，消除BRAM同步读取延迟

4. **局部历史分支预测单元**：加深BHT/PHT + BTB，ID阶段提前解析分支，误预测惩罚仅1周期

5. **M扩展运算单元**：2周期流水线乘法器 + 串行除法器（支持提前终止），按`XLEN`覆盖RV32M/RV64M

6. **冒险处理单元**：四级数据旁路（EX→EX、MEM→EX、WB→EX、MEM→MEM），Load-Use检测与1周期气泡插入

7. **CSR寄存器组**：Machine模式最小子集，支持计时、异常和可选中断入口

8. **存储子系统**：双端口IMEM + 双端口DMEM（FPGA原生BRAM），哈佛结构

9. **AXI4-Lite总线接口**：标准精简实现，CPU暂停期间允许调试访问

### 3.2 架构核心优化亮点

1. **分支提前解析**：分支判断移至ID阶段，误预测仅冲刷IF一级，惩罚降至1周期

2. **四级数据旁路**：覆盖所有常见数据依赖场景，最大化消除气泡

3. **增强预测**：局部历史分支预测器采用更深BHT/PHT/BTB，提高循环与嵌套分支预测准确率

4. **增强预取**：4级指令预取覆盖连续取指与预测目标取指，降低BRAM同步读取和预测恢复带来的前端空泡

5. **位宽自适应**：RV32I/RV64I共用一套控制逻辑，数据通路通过`XLEN`参数自动适配

6. **除法器提前终止**：检测操作数前导零，跳过无效迭代周期，平均缩短50%除法延迟

## 四、核心模块详细设计

### 4.1 指令预取模块（IF阶段增强）

#### 设计目标

BRAM为同步存储器，读取存在1周期延迟。预取模块的核心目的是隐藏该延迟，使流水线在大多数情况下单周期获取指令。

#### 架构实现

```
          ┌──────────────────────────────┐
          │       指令预取单元            │
          │                              │
  PC ────→│ ┌──────┐┌──────┐┌──────┐┌──────┐ │
          │ │Slot 0││Slot 1││Slot 2││Slot 3│ │──→ 指令输出
          │ │PC+4  ││PC+8  ││PC+12 ││PC+16 │ │
          │ └──────┘└──────┘└──────┘└──────┘ │
          │       ↑                      │
          │   IMEM Port A 读取          │
          │                              │
  预测 ──→│  预取地址选择逻辑            │
  目标    │  (顺序 / 分支目标)          │
          └──────────────────────────────┘
```

- 采用**4级指令预取缓存（Slot 0 ~ Slot 3）**，默认缓存PC+4、PC+8、PC+12、PC+16位置的顺序指令；预测跳转时优先填充目标地址及其后续顺序指令

- 每个Slot由1个32位指令寄存器 + 1个地址标签寄存器 + 1个有效位组成

- 与局部历史分支预测器联动，预测跳转时直接预取目标地址指令

- 纯寄存器实现，**不占用BRAM/DSP资源**。默认资源开销约为 4×(32位指令 + 32位地址 + 1位有效位) = 260位寄存器；RV64模式下指令字仍为32位，地址标签宽度随实现地址宽度参数化

#### 工作逻辑

1. **顺序预取**：当前PC执行时，预取单元通过IMEM端口A流水化读取PC+4、PC+8、PC+12、PC+16指令，依次填入Slot 0~3

2. **命中输出**：下一周期PC推进至PC+4时，直接从Slot 0输出指令，无需等待BRAM读取延迟

3. **分支联动**：分支预测有效时，清空当前Slot，转为预取跳转目标地址处的指令

4. **未命中处理**：预取地址与实际PC不匹配时（预测错误后），回退至正常BRAM读取，下一周期插入1个取指气泡后恢复预取

5. **冲刷恢复**：流水线冲刷时，清空所有Slot的有效位，从正确PC重新开始预取

### 4.2 局部历史分支预测器

#### 架构设计

```
                  ┌─────────────────────────────────────┐
  PC[9:2] ───────→│  BHT (Branch History Table)         │
  (索引)          │  128条目 × 6位局部历史               │
                  │                                     │
                  │  ┌───────┐     ┌──────────────┐     │
                  │  │ BHR   │────→│ PHT (2位饱和 │     │
                  │  │6位历史│     │ 计数器×64)   │     │──→ 预测结果
                  │  └───────┘     └──────────────┘     │   (跳转/不跳转)
                  └─────────────────────────────────────┘

  PC[12:2] ──────→│  BTB (Branch Target Buffer)         │
  (索引)          │  64条目 × (标签 + 目标地址)         │──→ 预测目标地址
                  └─────────────────────────────────────┘
```

- **BHT（分支历史表）**：`BHT_DEPTH`条目（默认128），每条目存储`BHR_WIDTH`位（默认6位）局部分支历史

- **PHT（模式历史表）**：每个BHT条目关联 2^BHR_WIDTH 个2位饱和计数器（默认每条目64个计数器），总计 128 × 64 × 2 = 16384位

- **BTB（分支目标缓存）**：`BTB_DEPTH`条目（默认64），每条目存储PC标签 + 目标地址，用于预测跳转目标

- 总资源占用：BHT 约768位 + PHT 约16384位 + BTB 约64×(20+32) = 3328位，合计约20480位（约2.5KB）寄存器/分布式RAM。Zynq-7020资源充足，默认优先使用更深预测表换取分支准确率

#### 分支提前解析机制

为实现1周期误预测惩罚，分支判断从EX阶段前移至ID阶段：

1. **ID阶段增加的硬件**：
   - 独立的分支比较器（BEQ/BNE/BLT/BGE/BLTU/BGEU），仅对寄存器堆读出值做比较，不复用ALU
   - 分支目标地址加法器（PC + 立即数偏移），独立于EX阶段ALU

2. **时序影响**：分支比较器为纯组合逻辑，32位比较在125MHz目标下仍有可优化空间；64位模式下需重点关注关键路径

3. **误预测流程**：
   - ID阶段解析出实际分支结果
   - 与预测结果比较，若不一致则产生冲刷信号
   - 仅需冲刷IF阶段1条指令，惩罚为**1周期**
   - 同时更新BHT/PHT/BTB

#### 预测流程

```
周期1 (IF): 用PC索引BHT/BTB → 获取预测结果与目标地址
            预测跳转: 下一周期从目标地址取指
            预测不跳转: 下一周期从PC+4取指

周期2 (ID): 分支指令译码 → 分支比较器计算实际结果
            预测正确: 流水线正常推进，无惩罚
            预测错误: 冲刷IF级，用正确地址重新取指（1周期惩罚）

周期3 (反馈): 更新BHT局部历史移位寄存器
              更新PHT 2位饱和计数器
              更新BTB目标地址（若为新分支或目标改变）
```

#### 2位饱和计数器状态机

```
             跳转           跳转           跳转
  强不跳转 ───────→ 弱不跳转 ───────→ 弱跳转 ───────→ 强跳转
  (00)     ←─────── (01)     ←─────── (10)   ←─────── (11)
            不跳转          不跳转          不跳转
```

- 预测输出：计数器高位为1时预测跳转，为0时预测不跳转
- 预期准确率：≥93%（CoreMark分支以循环为主，局部历史模式匹配度高，默认预测表加深后冲突率更低）

### 4.3 M扩展运算单元

#### 乘法器（2周期流水线实现）

- 采用**2级流水线乘法器**（`MUL_STAGES=2`），优先映射到Zynq-7020的DSP48资源以提高吞吐和时序裕量

- **第1周期**：计算低16位部分积与交叉部分积

- **第2周期**：累加部分积，输出最终64位结果（取高/低32位由指令决定）

- 资源选择策略：
  - **FPGA目标**：优先使用DSP48硬核（Xilinx）或嵌入式乘法器（Intel），单个DSP48可完成18×25位有符号乘法，2个DSP48即可覆盖32位乘法，不额外消耗LUT
  - **ASIC目标或无DSP时**：退化为纯LUT阵列乘法器，资源约2000 LUT

- 流水线乘法不阻塞后续指令：乘法结果在2周期后写回，编译器/硬件可调度无关指令填充

#### 除法器（串行迭代+提前终止）

- 采用**基4 SRT串行除法器**，每周期处理2位商，最大周期数 = `XLEN/2` = 16周期（RV32）

- **提前终止优化**（`DIV_EARLY_TERM=1`）：
  - 检测被除数前导零数量，跳过对应的无效迭代
  - 特殊值快速路径：除数为1时1周期完成，被除数为0时1周期返回0
  - 平均除法周期：约10周期（CoreMark典型数据分布）

- **流水线暂停机制**：
  - 除法指令进入EX阶段后，向冒险单元发出`div_busy`信号
  - 冒险单元暂停IF/ID/EX阶段的流水线寄存器，MEM/WB正常推进
  - 除法完成后释放`div_busy`，恢复流水线正常推进
  - 连续除法指令之间无需额外启动开销

- **对CoreMark性能影响估算**：
  - CoreMark中除法指令占比约1~3%
  - 平均额外开销：2% × 10周期 = 0.2 CPI增量
  - 对总IPC影响：约降低15%，在可接受范围内

#### 指令支持

RV32M支持：

| 指令 | 功能 | 延迟 |
|------|------|------|
| MUL | 低32位乘积 | 2周期 |
| MULH | 高32位（有符号×有符号） | 2周期 |
| MULHSU | 高32位（有符号×无符号） | 2周期 |
| MULHU | 高32位（无符号×无符号） | 2周期 |
| DIV | 有符号除法 | 2~16周期 |
| DIVU | 无符号除法 | 2~16周期 |
| REM | 有符号取余 | 2~16周期 |
| REMU | 无符号取余 | 2~16周期 |

当`XLEN=64`且`ENABLE_M_EXT=1`时，乘除法数据通路扩展至64位，并额外支持RV64M的MULW、DIVW、DIVUW、REMW、REMUW。W后缀指令仅取低32位结果并符号扩展至64位；对应验证需覆盖`rv64um-p-*`测试集。

### 4.4 5级流水线详细设计

#### 流水线结构与信号

```
  ┌──────┐    ┌──────┐    ┌──────┐    ┌──────┐    ┌──────┐
  │  IF  │───→│  ID  │───→│  EX  │───→│ MEM  │───→│  WB  │
  │      │    │      │    │      │    │      │    │      │
  │PC    │    │译码  │    │ALU   │    │DMEM  │    │写回  │
  │预取  │    │寄存器│    │乘除法│    │读写  │    │选择  │
  │IMEM  │    │读取  │    │旁路  │    │      │    │      │
  │      │    │分支  │    │      │    │      │    │      │
  │      │    │解析  │    │      │    │      │    │      │
  └──┬───┘    └──┬───┘    └──┬───┘    └──┬───┘    └──┬───┘
     │           │           │           │           │
  ┌──┴───────────┴───────────┴───────────┴───────────┴──┐
  │              流水线寄存器 (IF/ID, ID/EX, EX/MEM, MEM/WB)  │
  └─────────────────────────────────────────────────────┘
```

#### 各级详细功能

**1. IF取指（含预取）**

- 指令预取单元发出预取地址至IMEM端口A
- 分支预测器查询BHT/BTB，确定下一PC
- PC多路选择：顺序PC+4 / 预测跳转目标 / 冲刷恢复地址
- IMEM双端口BRAM：端口A供CPU取指/预取，端口B供AXI4-Lite调试写入
- 输出：32位指令字 + 当前PC值 → IF/ID流水线寄存器

**2. ID译码与分支解析**

- 指令译码器：RV32I基础指令（37条）+ RV32M扩展（8条）+ CSR指令（6条），RV32IM配置共51条指令
- 当`XLEN=64`时额外译码：ADDIW/SLLIW/SRLIW/SRAIW/ADDW/SUBW/SLLW/SRLW/SRAW/LWU/LD/SD等12条RV64I W/D后缀指令；启用M扩展时再增加MULW/DIVW/DIVUW/REMW/REMUW等5条RV64M指令
- 寄存器堆读取：双端口读，支持同周期读两个源操作数
- 立即数生成：I/S/B/U/J五种格式立即数提取与符号扩展
- **分支比较器**：独立于ALU，在ID阶段完成分支条件判断
- **分支目标加法器**：独立计算PC + imm_B / imm_J
- 控制信号生成：ALU操作码、访存控制、写回选择、CSR操作类型
- 输出：控制信号 + 源操作数 + 立即数 + PC → ID/EX流水线寄存器

**3. EX执行**

- **ALU**：加减/逻辑/移位/比较，支持RV32I全部运算
- **乘法器**：2周期流水线，第1周期部分积计算，第2周期累加输出
- **除法器**：串行迭代，`div_busy`信号控制流水线暂停
- **数据旁路输入**：接收来自MEM级、WB级的转发数据
- **结果选择**：ALU结果 / 乘法结果 / PC+4（JAL/JALR）
- 输出：运算结果 + 访存数据 + 控制信号 → EX/MEM流水线寄存器

**4. MEM访存**

- DMEM读写：单周期访问，支持LB/LH/LW/LBU/LHU（RV64I额外支持LWU/LD/SD）
- 地址对齐检查：非对齐访问触发异常
- 字节/半字/字选择与符号扩展
- **Store数据旁路**：接收来自WB级的转发数据，处理store-after-load依赖
- CSR读写：在此阶段执行CSR指令，避免与EX阶段竞争
- 输出：访存数据 / ALU直通结果 → MEM/WB流水线寄存器

**5. WB写回**

- 写回数据选择：ALU结果 / 访存数据 / CSR读取值 / PC+4
- 单写入端口写回寄存器堆
- 写回使能由控制信号决定，x0寄存器硬连线为0不可写

#### 流水线关键时序场景

**场景1：正常顺序执行（IPC=1）**

```
周期:    1    2    3    4    5    6    7
Inst1:  IF   ID   EX  MEM   WB
Inst2:       IF   ID   EX  MEM   WB
Inst3:            IF   ID   EX  MEM   WB
```

**场景2：Load-Use数据冒险（插入1气泡）**

```
周期:    1    2    3    4    5    6    7    8
LW x1:  IF   ID   EX  MEM   WB
ADD x1:      IF   ID  stall  EX  MEM   WB        ← 检测到Load-Use，ID阶段插入气泡
Inst3:            IF  stall  ID   EX  MEM   WB
```

**场景3：分支预测正确（无惩罚）**

```
周期:    1    2    3    4    5    6
BEQ:    IF   ID   EX  MEM   WB           ← ID阶段解析分支，预测正确
目标:        IF   ID   EX  MEM   WB      ← 预测命中，无气泡
```

**场景4：分支预测错误（1周期惩罚）**

```
周期:    1    2    3    4    5    6    7
BEQ:    IF   ID   EX  MEM   WB           ← ID阶段解析分支，预测错误
错误:        IF   ×                       ← IF级指令被冲刷
正确:             IF   ID   EX  MEM   WB  ← 从正确地址重新取指
```

**场景5：除法指令（流水线暂停）**

```
周期:    1    2    3    4    ...  N+3  N+4  N+5
DIV:    IF   ID   EX   EX  ...  EX   MEM   WB    ← EX阶段占用N个周期
Next:        IF   ID  stall ... stall  ID   EX    ← IF/ID暂停等待除法完成
```

### 4.5 冒险处理单元

#### 4.5.1 数据旁路网络

```
                    ┌─────────────────────────┐
                    │     旁路选择逻辑         │
                    │                         │
  WB级写回数据 ────→│  WB→EX (2周期间隔依赖)  │───→ EX级源操作数A
  MEM级ALU结果 ───→│  MEM→EX (1周期间隔依赖) │───→ EX级源操作数B
  EX级ALU结果 ────→│  EX→EX  (寄存器堆旁路)  │
                    │                         │
  WB级写回数据 ────→│  MEM→MEM (Store旁路)    │───→ MEM级Store数据
                    └─────────────────────────┘
```

四条旁路通路：

| 旁路路径 | 场景 | 实现方式 |
|----------|------|----------|
| EX→EX | 连续两条指令，后者依赖前者ALU结果 | EX/MEM寄存器输出直接反馈至EX级输入MUX |
| MEM→EX | 间隔1条指令的数据依赖 | MEM/WB寄存器输出反馈至EX级输入MUX |
| WB→EX | 间隔2条指令的数据依赖 | WB级写回数据反馈至EX级输入MUX |
| MEM→MEM | Store指令数据依赖前序Load结果 | MEM/WB寄存器输出转发至DMEM写数据端口 |

旁路优先级：EX→EX > MEM→EX > WB→EX（最新结果优先）

#### 4.5.2 Load-Use冒险检测

```verilog
// Load-Use检测逻辑（ID阶段）
wire load_use_hazard = id_ex_mem_read              // 前序指令为Load
                     & (id_ex_rd != 5'b0)          // 目标寄存器非x0
                     & ( (id_ex_rd == if_id_rs1)   // 当前指令rs1依赖
                       | (id_ex_rd == if_id_rs2)); // 当前指令rs2依赖
```

检测到Load-Use时：
- IF/ID流水线寄存器保持不变（暂停）
- PC保持不变（暂停取指）
- ID/EX流水线寄存器插入NOP气泡（控制信号清零）
- 下一周期Load数据到达MEM级，通过MEM→EX旁路转发

#### 4.5.3 控制冒险处理

- 分支预测错误：ID阶段检测到预测错误后，发出`flush_if`信号，清空IF/ID流水线寄存器
- JAL/JALR指令：在ID阶段计算跳转目标，1周期惩罚（与分支处理相同）
- 异常/中断：清空IF/ID和ID/EX流水线寄存器，PC跳转至`mtvec`

#### 4.5.4 结构冒险

- 哈佛存储结构，IMEM与DMEM物理分离，取指与访存无资源冲突
- 寄存器堆采用"写前读"（Write-First）策略，同周期同地址读写时读取新值
- 除法器占用期间不产生结构冒险（通过流水线暂停避免）

### 4.6 CSR寄存器组（Machine模式最小子集）

#### 支持的CSR列表

| CSR地址 | 名称 | 读写 | 说明 |
|---------|------|------|------|
| 0x300 | mstatus | RW | 机器状态寄存器（MIE/MPIE/MPP字段） |
| 0x301 | misa | RO | ISA描述寄存器（按`XLEN`和`ENABLE_M_EXT`报告RV32I/RV32IM/RV64I/RV64IM） |
| 0x304 | mie | RW | 机器中断使能（MTIE/MEIE/MSIE） |
| 0x305 | mtvec | RW | 异常/中断入口地址（仅支持Direct模式） |
| 0x340 | mscratch | RW | 机器模式暂存寄存器 |
| 0x341 | mepc | RW | 异常返回地址 |
| 0x342 | mcause | RW | 异常/中断原因编码 |
| 0x343 | mtval | RW | 异常附加信息（故障地址/指令） |
| 0x344 | mip | RO | 机器中断挂起状态 |
| 0xB00 | mcycle | RO | 周期计数器（RV32为低32位，RV64为完整64位） |
| 0xB02 | minstret | RO | 已退休指令计数器（RV32为低32位，RV64为完整64位） |
| 0xB80 | mcycleh | RO | 周期计数器高32位（仅RV32） |
| 0xB82 | minstreth | RO | 已退休指令计数器高32位（仅RV32） |
| 0xF11 | mvendorid | RO | 厂商ID（可配置） |
| 0xF12 | marchid | RO | 架构ID（可配置） |
| 0xF13 | mimpid | RO | 实现ID（可配置） |
| 0xF14 | mhartid | RO | 硬件线程ID（= `HART_ID`参数） |

#### CSR指令支持

支持6条CSR操作指令：CSRRW、CSRRS、CSRRC、CSRRWI、CSRRSI、CSRRCI

#### 中断支持边界

默认最小配置只保证异常处理和性能计数器可用，`mie/mip`字段作为Machine模式兼容接口保留。若启用中断输入，则支持Machine软件中断、Machine定时器中断和Machine外部中断三类入口，触发条件为`mstatus.MIE=1`且对应`mie`与`mip`位同时有效；进入中断时复用异常入口流程，`mcause`最高位标记为中断，低位记录中断原因编码。

#### CoreMark计时机制

CoreMark在RV32模式下通过读取`mcycle`/`mcycleh`组合获得64位周期计数，在RV64模式下直接读取64位`mcycle`，计算执行时间。实现要点：
- `mcycle`每个时钟周期自增1；RV32模式下低32位溢出时`mcycleh`进位，RV64模式下`mcycle`自身为64位计数器
- `minstret`每退休一条指令自增1（流水线冲刷/气泡的指令不计数）
- RV32读取时先读`mcycleh`，再读`mcycle`，再读`mcycleh`验证是否溢出翻转；RV64可直接读取`mcycle`

### 4.7 异常处理机制

#### 支持的异常类型

| mcause编码 | 异常类型 | 触发条件 |
|------------|----------|----------|
| 0 | 指令地址未对齐 | 分支/跳转目标非4字节对齐 |
| 1 | 指令访问错误 | PC超出IMEM地址范围 |
| 2 | 非法指令 | 译码失败或不支持的指令编码 |
| 4 | 取数地址未对齐 | Load地址不满足自然对齐 |
| 5 | 取数访问错误 | Load地址超出DMEM范围 |
| 6 | 存数地址未对齐 | Store地址不满足自然对齐 |
| 7 | 存数访问错误 | Store地址超出DMEM范围 |
| 11 | 环境调用（ECALL） | 执行ECALL指令 |
| 3 | 断点（EBREAK） | 执行EBREAK指令 |

#### 异常处理流程

```
1. 异常检测（ID/EX/MEM阶段均可产生异常）
2. 记录异常信息：
   - mepc ← 异常指令PC
   - mcause ← 异常原因编码
   - mtval ← 故障地址或非法指令编码
   - mstatus.MPIE ← mstatus.MIE（保存中断使能状态）
   - mstatus.MIE ← 0（关闭中断）
3. 冲刷流水线（清空异常指令之后的所有流水级）
4. PC ← mtvec（跳转至异常处理入口）

MRET指令返回流程：
1. PC ← mepc（恢复异常前PC）
2. mstatus.MIE ← mstatus.MPIE（恢复中断使能）
3. mstatus.MPIE ← 1
```

#### 异常优先级

同一指令触发多个异常时，优先级从高到低：指令地址异常 > 非法指令 > 数据地址异常

### 4.8 FENCE指令处理

RV32I基础指令集包含FENCE指令，在哈佛结构下的实现策略：

- **FENCE指令**：由于本设计为单核顺序执行、无缓存（直连BRAM），访存天然有序，FENCE指令作为NOP处理，不产生额外流水线停顿

- **FENCE.I指令**（若支持Zifencei扩展）：
  - 冲刷整条流水线
  - 清空指令预取缓存所有Slot
  - 确保此前通过DMEM写入的指令对后续取指可见
  - 由于IMEM/DMEM物理分离，自修改代码需通过AXI4-Lite接口将修改后的指令写入IMEM，不支持CPU侧直接修改指令存储

### 4.9 存储子系统

#### 指令存储器（IMEM）

- **类型**：双端口BRAM（True Dual Port）
- **默认容量**：`IMEM_DEPTH` × 4 = 64KB
- **端口A**：CPU取指/预取，只读，同步读取（1周期延迟，由预取模块隐藏）
- **端口B**：AXI4-Lite调试写入，CPU运行时只写、CPU暂停时可读写
- **初始化**：支持`$readmemh`从hex文件加载初始指令
- **资源占用**：约16块BRAM36K（64KB配置）

#### 数据存储器（DMEM）

- **类型**：双端口BRAM（True Dual Port）
- **默认容量**：`DMEM_DEPTH` × 4 = 32KB
- **端口A**：CPU访存，读写，单周期访问
- **端口B**：AXI4-Lite调试访问
- **端口仲裁策略**：
  - CPU运行期间，端口A优先，AXI4-Lite访问端口B等待CPU空闲周期（CPU不发起访存的周期即可通过端口B访问）
  - CPU暂停期间（如debug halt），AXI4-Lite独占两个端口
  - 双端口BRAM天然支持两端口同时访问不同地址，因此大多数情况下无需仲裁
- **字节写使能**：支持SB/SH/SW的字节粒度写入
- **资源占用**：约8块BRAM36K（32KB配置）

CoreMark默认建议采用IMEM 64KB + DMEM 32KB配置。若链接完整newlib/printf、加入较复杂的板级输出代码，或扩大CoreMark数据集，可将`DMEM_DEPTH`提升至16384，形成IMEM 64KB + DMEM 64KB配置。

#### 资源汇总

| 存储模块 | BRAM36K数量 | 容量 |
|----------|-------------|------|
| IMEM | 16 | 64KB |
| DMEM | 8 | 32KB |
| **合计** | **24** | **96KB** |

### 4.10 AXI4-Lite总线接口

#### 通道信号

严格遵循AXI4-Lite协议5个核心通道：

| 通道 | 方向 | 核心信号 |
|------|------|----------|
| AW（写地址） | Master→Slave | AWADDR, AWVALID, AWREADY, AWPROT |
| W（写数据） | Master→Slave | WDATA, WSTRB, WVALID, WREADY |
| B（写响应） | Slave→Master | BRESP, BVALID, BREADY |
| AR（读地址） | Master→Slave | ARADDR, ARVALID, ARREADY, ARPROT |
| R（读数据） | Slave→Master | RDATA, RRESP, RVALID, RREADY |

#### 地址映射

| 地址范围 | 目标模块 | 说明 |
|----------|----------|------|
| 0x0000_0000 ~ 0x0000_FFFF | IMEM | 指令存储器（64KB） |
| 0x0001_0000 ~ 0x0001_7FFF | DMEM | 数据存储器（32KB） |
| 0x0002_0000 ~ 0x0002_00FF | CSR | CSR寄存器映射（调试用） |

#### CPU运行/暂停控制

- AXI4-Lite主设备（上位机）通过控制寄存器发送暂停/恢复/复位命令
- 暂停状态下CPU流水线冻结，AXI4-Lite可自由读写IMEM/DMEM
- 运行状态下AXI4-Lite仅可通过端口B访问存储器

#### 程序加载流程

```
1. 上位机发送CPU暂停命令
2. 通过AXI4-Lite写通道将程序二进制写入IMEM地址空间
3. 可选：初始化DMEM数据段
4. 上位机发送CPU复位+启动命令
5. CPU从RESET_ADDR开始执行
```

定位：资源预算明显低于Zynq-7020容量，后续实现可继续使用更多DSP/BRAM/LUT换取更高主频、更深预测表或更大的片上存储。

### 5.3 时序关键路径分析

| 关键路径 | 预期延迟 | 125MHz裕量 | 优化策略 |
|----------|----------|------------|----------|
| ID级分支比较器 | ~6ns | 2ns | 32位比较器纯组合逻辑，必要时对ID级控制路径做局部寄存 |
| EX级ALU加法器 | ~5ns | 3ns | 超前进位加法器，保持ALU输入MUX级数受控 |
| 旁路MUX→ALU→结果 | ~7ns | 1ns | 关键路径，需确保MUX级数≤2，必要时拆分复杂操作 |
| BRAM读取 | ~2ns | 6ns | BRAM原语内置，延迟固定 |

64位模式下（`XLEN=64`），分支比较器和ALU关键路径延迟约增加2~3ns。RV64IM以100MHz保底为主，125MHz作为优化目标，需要根据综合后时序报告决定是否增加局部流水或降低旁路MUX复杂度。

## 七、Verilog设计规范

1. 全参数化设计，所有可配置参数集中顶层`cpu_top`模块，各子模块通过参数传递继承

2. 组合逻辑与时序逻辑分离，避免`always @(*)`块中出现不完整条件导致锁存器

3. 模块功能单一化，逻辑复用最大化，杜绝重复代码

4. 信号命名规范：
   - 流水线级前缀：`if_`、`id_`、`ex_`、`mem_`、`wb_`
   - 流水线寄存器：`if_id_xxx`、`id_ex_xxx`、`ex_mem_xxx`、`mem_wb_xxx`
   - 控制信号：`ctrl_xxx`
   - 有效标志：`xxx_valid`
   - 使能信号：`xxx_en`

5. 流水线寄存器按需设置，仅传递下游级实际使用的信号

6. 复位策略：同步复位，所有流水线寄存器和控制状态在复位时清零

7. 端口声明使用`input wire`/`output reg`/`output wire`显式声明

8. 综合约束：不使用`initial`块（仅用于仿真初始化）、不使用`#delay`

## 八、方案优势总结

1. **性能优先**：基于IPC损耗量化分析，100MHz下CoreMark≥230Iter/s、125MHz优化目标≥285Iter/s有明确架构支撑

2. **适配Zynq-7020**：默认使用更深预测、更大片上存储和DSP优先乘法器，资源预算仍明显低于Zynq-7020容量

3. **完整指令集**：RV32I/RV64I可切换，M扩展可选，RV32M全8条和RV64M W后缀乘除法指令硬件实现

4. **分支1周期惩罚**：分支提前至ID阶段解析 + 局部历史预测，分支性能损耗最小化

5. **完整系统支持**：Machine模式CSR + 异常处理 + 性能计数器，满足riscv-tests与CoreMark全部系统需求

6. **标准总线兼容**：AXI4-Lite精简接口，完整地址映射与仲裁策略，无缝接入SoC

7. **可扩展性能配置**：顶层参数集中管理，可继续提高预测深度、预取深度、存储大小或DSP使用量以换取性能
